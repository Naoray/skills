# Comprehension Principles

C1. A review artifact must expose runtime behavior, not restate a diff.

C2. Every diagram needs an entry point, terminal outcome, and visible error path.

C3. Use domain language in labels. Avoid code identifiers unless they are the
only precise name.

C4. Show what changed. Mark changed steps so the reader can separate new behavior
from existing behavior.

C5. Prefer one useful diagram over a complete but unreadable map.

C6. State scope gaps. A reviewer must know what the artifact does not prove.
