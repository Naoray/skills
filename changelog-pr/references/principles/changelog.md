# Changelog Principles

C1. Use Keep a Changelog categories: Added, Changed, Deprecated, Removed, Fixed,
Security.

C2. Write for users, operators, and reviewers. Do not dump raw commit subjects.

C3. Include why the change matters when the title alone is ambiguous.

C4. Group related commits into one entry when separate entries would repeat the
same user-visible change.

C5. Omit internal-only changes unless they affect behavior, operations, security,
compatibility, migration, or debugging.

C6. In squash-merge repositories, use merged PR metadata and previous release PR
coverage. SHA ancestry alone can over-report already shipped work.

C7. Preserve manual release notes unless clearly stale.
